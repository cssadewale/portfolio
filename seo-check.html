📋 All Known Causes — With Exact Fixes

Google's "Couldn't fetch" error has exactly 7 possible causes. Below is every one, ranked by how likely it is for a Cloudflare Pages site like yours, with exact fix steps for each.

1. sitemap.xml was not actually uploaded to GitHub MOST LIKELY
"Couldn't fetch" most commonly means Google tried to download sitemap.xml and received a 404 — the file does not exist at the URL you submitted. Google does not distinguish between "file not found" and "server error" in this error message.

How to fix it

1
Open a new browser tab and go to: https://cssadewale.pages.dev/sitemap.xml
2
If you see XML code with your page URLs → the file is uploaded correctly. Skip to cause #2.
If you see "404 Not Found" or any error page → the file was not uploaded.
3
If not uploaded: Go to github.com/cssadewale → open your portfolio repository → click "Add file" → "Upload files" → upload sitemap.xml and robots.txt → commit.
4
Wait 60 seconds for Cloudflare Pages to rebuild → open https://cssadewale.pages.dev/sitemap.xml again → confirm you see XML.
5
Go back to Google Search Console → Sitemaps → click on your sitemap → click "Resubmit".
2. You submitted the wrong URL in Search Console VERY LIKELY
When Search Console asks for your sitemap, it shows your domain and asks you to type the path. Many people accidentally type the full URL instead of just the path, or type it incorrectly.

Check exactly what you submitted

1
In Search Console, go to Indexing → Sitemaps.
2
Look at the URL shown next to "Couldn't fetch". It should read exactly: https://cssadewale.pages.dev/sitemap.xml
3
If it shows anything different (e.g., sitemap.xml/sitemap.xml or https://https://...), delete it and resubmit correctly.
4
To resubmit correctly: In the "Add a new sitemap" box, Search Console shows your domain. You type ONLY: sitemap.xml — not the full URL, just the filename.
3. Google has not tried yet — it is still queued COMMON — BE PATIENT
When you first submit a sitemap, Google shows "Couldn't fetch" as a placeholder status before it actually tries to crawl it. This is confusing but normal. Google may not attempt the crawl for 24–72 hours after submission, especially for new sites.

How to tell if this is the reason

1
Check the "Last read" column next to your sitemap in Search Console.
If it shows "—" or "Never" → Google has not tried yet. Wait 48 hours and check again.
2
If it shows a date and still says "Couldn't fetch" → Google did try and failed. This means causes 1, 2, or 4 apply.
3
To force Google to try sooner: go to the URL Inspection Tool in Search Console → type https://cssadewale.pages.dev/sitemap.xml → click "Request Indexing".
4. robots.txt is blocking Googlebot from the sitemap CHECK THIS
If your robots.txt file has a Disallow: /sitemap.xml rule (even accidentally), Google will respect it and refuse to fetch the sitemap, reporting "Couldn't fetch".

How to check and fix

1
Open: https://cssadewale.pages.dev/robots.txt in your browser.
2
Confirm it contains exactly this (no extra Disallow lines that mention sitemap):
User-agent: *
Allow: /
Disallow: /admin.html
Disallow: /github-fix.html
Sitemap: https://cssadewale.pages.dev/sitemap.xml
3
If robots.txt does not exist (404 error) → you did not upload it. Upload it to GitHub root and redeploy.
5. Search Console property does not match your sitemap domain CHECK IF OTHERS FAIL
If you added your site to Search Console as https://cssadewale.pages.dev (URL prefix) but your sitemap says http://cssadewale.pages.dev (no S), or if you added it as a Domain property with a different URL format, Google will reject the sitemap.

How to verify

1
In Search Console, look at the top left where your property name is shown.
2
It must show: https://cssadewale.pages.dev/ — with HTTPS and the trailing slash.
3
If it shows http:// or a different domain, delete the property and re-add it as https://cssadewale.pages.dev/.
6. Cloudflare Pages Bot Fight Mode blocking Googlebot RARE FOR FREE TIER
Cloudflare has a "Bot Fight Mode" that can mistakenly block legitimate crawlers including Googlebot. This is more common on paid Cloudflare plans with WAF rules enabled. The free Cloudflare Pages plan rarely has this issue.

How to check

1
This only affects you if you have a custom domain with Cloudflare DNS pointing to your Pages site. If you are purely on cssadewale.pages.dev, this does not apply.
2
If you have a custom domain: Log into dash.cloudflare.com → Security → Bots → disable "Bot Fight Mode".
7. Malformed XML in sitemap.xml UNLIKELY — FILE IS VALID
The sitemap.xml file provided for your site was tested and confirmed as valid XML. However, if you edited the file manually or it got corrupted during upload, the XML could be malformed.

How to verify

1
Go to xml-sitemaps.com/validate-xml-sitemap.html
2
Enter your sitemap URL: https://cssadewale.pages.dev/sitemap.xml and click Validate.
3
If it shows errors, re-download the clean sitemap.xml file from the portfolio package and re-upload it to GitHub.
✅ The Method That Works Regardless of Sitemap Status

Here is the honest truth: Google does not need your sitemap to index your pages. The sitemap is helpful but optional. You can get every page indexed right now using the URL Inspection Tool, even while the sitemap error persists.

Request indexing for each page manually — do this NOW

1
Go to search.google.com/search-console/
2
In the search bar at the very top of Search Console, type: https://cssadewale.pages.dev/ → press Enter.
3
If status says "URL is not on Google" → click the big button "Request Indexing" → wait for confirmation popup.
4
Repeat for each page — paste these one at a time:
https://cssadewale.pages.dev/about.html
https://cssadewale.pages.dev/projects.html
https://cssadewale.pages.dev/journey.html
https://cssadewale.pages.dev/cv.html
5
This directly tells Google to crawl each page. Google usually processes indexing requests within 24–72 hours. The sitemap error does not affect this at all.
